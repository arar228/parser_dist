1. Общие положения 
Цель документа  
Для оперативного создания и изменения заказов, а также получения информации об их 
состоянии  необходимо настроить  взаимодействие между учетными системами Ресурс-Медиа (РМ) 
и покупателей (в дальнейшем - Пользователь). В предлагаемом документе описан порядок настроек 
таких систем с помощью операций API. 
Описание 
Обмен данными производится с помощью XML-сообщений. Сервис работает по протоколу 
SOAP, с помощью которого реализован удаленный вызов процедур, и позволяет выполнять 
удаленные запросы к базе данных Ресурс-Медиа для модификации заказов, получения информации 
о состоянии заказов, остатках товаров, характеристик номенклатуры и проч. Работа сервиса 
осуществляется в целях обеспечения безопасности по протоколу HTTPS. 
Методические требования к работе с данными 
Алгоритмы запросов к базе РМ должны быть построены так, чтобы минимизировать 
количество запросов. Для этого редко изменяемые данные, запрашиваемые Пользователем из базы 
РМ, должны храниться в его базе. Например, результаты, возвращаемые операциями 
GetMaterialData (для одних и тех же номенклатур), GetClientsAvail, GetWareHouses и др. будут 
практически всегда одинаковые.  
Предполагаемая методика использования API для получения цен и остатков: 
1. Получить GetCatalog - сохранить в своей базе. 
2. Получить GetMaterialData по всем интересующим товарным группам - сохранить в своей базе 
все необходимые для вашей работы поля. 
3. Если по какой-то причине не используются операции GetPrices и GetItemsAvail, но очень 
критично появление новинок, то данные из предыдущих пунктов обновлять периодически, 
например, каждые 2 часа. Иначе – см. далее. 
4. По необходимости вызывать GetPrices для каждого склада (с ограничивающими параметрами 
или без), связать результат по MaterialID с хранимой у себя таблицей из п.2 - получить всю 
необходимую информацию о номенклатуре.

5. Если в пункте 4 вернулись номенклатуры, которые отсутствуют в сохраненной у вас таблице

Общее описание методики работы с заказом  
Заказ создается операцией SetOrderCreate. После этого его можно изменять произвольное 
количество раз до наступления события подписи или финализации заказа. Для изменения 
используется операция SetOrderChange. Для того, чтобы заказ был принят в работу менеджерами 
РМ или для самостоятельной передачи заказа в набор, необходимо выполнить операции подписи 
(необязательна) и финализации. Подпись выполняется операцией SetSignOrder. После выполнения 
этой операции в базе РМ у заказа появляется признак того, что он подписан и не может быть изменен 
операцией SetOrderChange. Последним этапом выполняется SetFinalOrder, которая может 
разбить изначальный заказ на несколько (например, при необходимости доставки частями). Также 
финализация может производиться без предварительной подписи заказа. После выполнения этой 
операции у заказа устанавливается соответствующий признак, и применение к нему любых 
операций Set* становится недоступным. Однако в любой момент можно читать данные по заказу 
операциями GetOrderHeader и GetOrderItems. 
Примечание. SetSignOrder введена для схемы с 
двухступенчатым утверждением заказа на стороне 
Пользователя, когда разные ответственные лица занимаются 
утверждением заказа и его дальнейшим продвижением по 
бизнесс-процессам. Если такая схема не применяется, то 
использование этой операции нецелесообразно. 
Операции Set* при своем вызове помещаются в общую очередь команд и в ответе возвращают 
идентификатор, с которым они зарегистрировались в очереди. Обработка очереди происходит через 
определенные промежутки времени (например, каждые 10 сек). Результат выполнения операций 
Set* можно получить операцией GetOperationResult. 
Важно! Не делайте вызов GetOperationResult просто в

2. Ограничения вызовов операций API 
2.1. 
Подсистема управления интервалами между вызовами 
одной и той же операции 
Данная подсистема предназначена для регулирования нагрузки на нашу учетную систему. 
Через неё проходят абсолютно все операции API в момент их вызова. Результат её работы 
проявляется в виде разрешения или запрещения дальнейшего выполнения алгоритма вызываемой 
операции. Если подсистема определила, что данный вызов запрещен, то в поле Result любой 
операции будет установлено значение 3, а в поле ErrorMessage – текст описания причины отказа. 
Разработчикам, осуществляющим интеграцию пользователей с нашим API, необходимо обеспечить 
трансляцию этих сообщений ответственным лицам пользователей для понимания ими причин 
отказа и корректировки своих действий.  
На текущий момент реализован контроль паузы между вызовами. В дальнейшем не 
исключена реализация контроля и других параметров. Рассмотрим примеры. 
Пример 1. Пользователь осуществляет вызов GetMaterialData с отбором по MaterailID и 
передает туда список из 500 позиций. Если подсистема определит, что данный вызов делается чаще 
разрешенного интервала, то пользователь получит: Result = 3, ErrorMessage = «Разрешенный 
интервал вызова GetMaterialData: 1800 сек» (сообщение может отличаться) Поэтому мы 
рекомендуем организовывать интеграцию таким образом, чтобы у конечного Пользователя (или 
алгоритма, если интеграция предполагает полную автоматизацию) была возможность реагировать 
на ошибку 3 изменением своего поведения. 
Пример 2. Пользователь осуществил вызов операции SetOrderCreate. При этом 
непосредственно вызов вернул Result = 0, т.е. он прошел контроль, и она помещена в очередь. После 
этого Пользователь пытается получить результат этой операции вызовом GetOperationResult. 
Предположим, Пользователь не учел, что данная операция также может иметь ограничения по 
частоте вызовов и делает два вызова с интервалом, например, 1 сек. В этом случае первый вызов 
вернет Result = 2, что говорит о том, что операция SetOrderCreate находится в обработке, второй же 
вызов вернет Result = 3 и  ErrorMessage что-то вроде «Разрешенный интервал вызова 
GetOperationResult: 9 сек», и означает это, что отказ в доступе получил именно GetOperationResult, 
а не SetOrderCreate. 
Выбор момента времени для вычисления интервала между вызовами одной и той же операции 
может осуществляться двумя способами:  
Ресурс-Медиа © 2024 
7 
1. от последнего разрешенного вызова;  
2. просто от последнего вызова.  
В первом случае операция получит разрешение по истечении заданного интервала независимо 
от количества вызовов, которые попытался сделать Пользователь за этот интервал. Во втором 
случае отсчет начинается заново от последней попытки вызова, т.о. если пользователь игнорирует 
сообщение о неистечении интервала и продолжит делать попытки вызова операции, разрешение он 
не получит «никогда». 
Пример. Вызов GetPrices был осуществлен в 14:00 и получил разрешение (вернул прайс). В 
14:10 была осуществлена повторная попытка вызова и получен ответ: Result = 3, ErrorMessage = 
«Разрешенный интервал вызова GetPrices: 3600 сек» (сообщение может отличаться). Если для 
правила, под которое попадает этот вызов, установлен контроль по первому способу, то сколько бы 
раз Пользователь ни пытался вызвать эту операцию, следующее разрешение он получит с 15:00. 
Если для этого правила контроль установлен по второму способу, то для ситуации, описанной в 
примере, отсчет начнется уже с 14:10 и если не будет ни одного вызова в течение 3600 сек, то 
следующее разрешение будет получено не ранее 15:10, если же Пользователь сделает вызов, 
скажем, в 14:40, то отсчет начнется именно с этого момента. 
Следует учесть, что правила доступа могут зависеть не только от имени операции, но и от 
передаваемых в нее параметров. Например, интервал между вызовами GetPrices тем больше, чем 
больше запрашиваемая в нем информация: по всему каталогу, по одной позиции, по тысяче и т.п. 
(цифры условные). 
Важно! Все результаты по получению разрешений 
фиксируются, и если будет выявлено систематическое 
игнорирование Пользователем предупреждений от 
данной системы, то его доступ к API может быть 
приостановлен.

SetFinalOrder 
Операция вызывается последней среди операций по модификации заказа и позволяет разбить 
один заказ на несколько. Заказы, полученные в результате разбиения, являются 
финализированными, поэтому их невозможно изменять с помощью API.  
Предусмотрен упрощенный вариант вызова этого метода – без указания OrderItems. В таком 
случае операция выполняется значительно быстрее, без модификации товарного состава заказа. 
Если нет необходимости на данном этапе изменить заказ или разбить его на несколько, 
рекомендуется использовать именно этот вариант. 
Если же передается таблица OrderItems, то обработка происходит следующим образом. Все 
переданные данные группируются по полю PurchaseOrderNumber. Если значений в этом поле будет 
больше одного, то исходный заказ будет разбит на количество переданных различных значений. 
Итоговый товарный состав всех получившихся заказов будет соответствовать содержащемуся в 
таблице OrderItems. Таким образом, на этом последнем этапе можно изменить товарный состав без 
предварительного использования операции SetOrderChange. 
Входные параметры 
DocumentNumber – номер заказа пользователя в базе РМ 
OrderItems – структура 
 Item – таблица позиций заказа 
o MaterialID – код номенклатуры 
o PurchaseOrderNumber – номер заказа в учетной системе Пользователя 
o Quantity – затребованное количество

4.7. 
Сервисные операции 
Notification 
Передает уведомления пользователю, которые РМ считает необходимым сообщить. Могут 
быть как общие, так и персональные, однако внешних различий в них нет. Уведомления имеют срок 
актуальности, после которого они перестают отправляться пользователю при вызове операции. В 
течение этого срока они будут отправляться при каждом вызове операции, т.е. нет возможности 
пометить их на стороне РМ как «прочитано». При передаче конкретной даты в соответствующий 
параметр операции можно получить все уведомления, которые были актуальны на указанную дату 
и до текущей. 
Важно! Реализация вызова данной операции обязательна 
для всех пользователей, поскольку уведомления, не 
предполагающие диалог в переписке, будут доводиться до 
сведения пользователей только с помощью нее (например, 
о предстоящем обновлении версии API, о планирующейся 
приостановке работы API и т.п.). Электронная почта в 
этих случаях использоваться не будет. 
Рекомендуемая методика использования данной операции. 
 Организовать вызов несколько раз в сутки. 
 При получении уведомления обеспечить передачу его содержимого ответственным 
лицам. 
 Т.к. уведомление может содержать файл вложения, сохранить его. 
 Поскольку уведомление имеет уникальный идентификатор, при необходимости 
реализовать в своей учетной системе механизм «прочитано». 
Входные параметры 
FromDate – дата, с которой необходимо получить публиковавшиеся уведомления. Если не указана, 
ищутся уведомления, актуальные на момент вызова.  
Выходные параметры 
Notification_Resp – структура  
 Notification – таблица уведомлений 
o NotificationID – уникальный идентификатор уведомления 
o Text – текст уведомления 
o Attachment – файл вложения, закодированный в формате base64Binary 
o AttachmentName – имя файла вложения 
 Result  
 ErrorMessage
