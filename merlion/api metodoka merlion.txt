1 ОСНОВНЫЕ СВЕДЕНИЯ 
3 
Web-сервис (API) B2B-модуля «Интернет-заказ» работает по протоколу SOAP (Simple Object Ac
cess Protocol) версии 1.1. поверх HTTPS. Обмен данными ведется посредством XML-сообщений, 
что позволяет реализовать удаленный вызов процедур (RPC - Remote Procedure Call). 
Для удобства работы с API размещается WSDL-описание (Web Services Description Language) 
версии 1.1. — это язык описания Web-сервисов, основанный на XML. На WSDL описываются ме
тоды, входные и результирующие структуры данных, типы данных, сетевые адреса для обра
щения к сервису и т.д. 
WSDL-описание сервиса (с различными стилями WSDL) доступно по ссылкам: 
 https://api.merlion.com/dl/mlservice2?wsdl (стиль document/literal) 
 https://api.merlion.com/re/mlservice2?wsdl (стиль rpc/encoded) 
 https://api.merlion.com/rl/mlservice2?wsdl (стиль rpc/literal) 
Для тестирования API WSDL-описание тестового сервиса доступно по ссылкам: 
 https://apitest.merlion.com/dl/mlservice2?wsdl (стиль document/literal) 
 https://apitest.merlion.com/re/mlservice2?wsdl (стиль rpc/encoded) 
 https://apitest.merlion.com/rl/mlservice2?wsdl (стиль rpc/literal) 
По WSDL-описанию можно построить клиентский класс API, используя любую SOAP-библиотеку. 
Библиотека скачивает WSDL-описание, анализирует и формирует структуры данных, необходи
мые для вызова методов. Приложению остается заполнить структуры данными и выполнить за
прос. Информация из ответного XML-сообщения выдается в виде структуры, свойственной язы
ку программирования. Все это избавляет приложение от необходимости обрабатывать XML. Су
ществует возможность самостоятельно формировать SOAP-запросы в формате XML и разбирать 
ответы, однако это трудоемкая задача, и работу с WSDL вручную выполнять нецелесообразно. 
Взаимодействие между серверами B2B-модуля «Интернет-заказ» и партнера осуществляется с 
помощью API. 
Сервис позволяет выполнять удаленные запросы к базе данных компании MERLION и получать 
актуальные данные, выполнять обновление в режиме он-лайн только необходимой части дан
ных, в отличие от скачивания полного прайс-листа в формате XML, формируемого несколько 
раз в день.  
API позволяет:  
 Автоматически резервировать в системе товар на складе компании MERLION, а также по
лучать информацию по созданным и ранее отгруженным заказам; 
 Подписывать заказ на отгрузку без участия менеджера компании MERLION (при соблю
дении общих и финансовых условий); 
 Импортировать в учетную систему актуальные цены и доступное для резерва количество 
товаров на складе по интересуемым товарным группам; 
 Использовать функциональность сервиса в интернет-магазинах и других торговых пло
щадках для расширения товарного ассортимента и наполнения товарных карточек ха
рактеристиками и изображениями товаров. 
Работа с API осуществляется по HTTPS (Hypertext Transfer Protocol Secure) протоколу с исполь
зованием авторизационных данных B2B-модуля «Интернет-заказ». Параметры подключения к 
сервису зависят от используемого протокола. Подробную информацию и примеры подключения 
к API смотрите в разделе «Подключение к API».

3.1 ПРАЙС-ЛИСТ 
6 
Прайс-лист можно получить с помощью трех отдельных методов: 
 getCatalog() – для получения справочника товарных групп;  
 getItems() – для получения справочника товаров;  
 getItemsAvail() – для получения доступного количества товаров и цен. 
Такой подход позволяет получать актуальные данные по всему прайс-листу или только по ин
тересуемым товарным группам, обращаясь к каждому методу с разной периодичностью. 
Классификация в товарном справочнике построена по принципу: 
Направления ⇒ Группы ⇒ Подгруппы ⇒ Товары 
Направления содержат группы, группы содержат подгруппы, подгруппы содержат товары. То
вары могут находиться только в подгруппах. 
Для направлений, групп и подгрупп, содержащих в коде букву «А», она указывается кирилли
цей. 
На расчет доступного количества и цен влияют значения метода отгрузки и даты отгрузки. В 
зависимости от метода отгрузки определяется, по какому складу и по какому прайс-листу будет 
произведен расчет данных. От даты отгрузки зависит доступное количество товаров для резер
ва. 
ВАЖНО! При выборе доставочного метода отгрузки (например, "ДОСТАВКА") необходимо ука
зывать дату отгрузки, начиная с завтрашнего дня, т.к. на сегодня доставка не осуществляется 
(при этом доступное количество товара будет нулевое). 
Все данные по количеству товара возвращаются в штуках. 
Пример получения прайс-листа: 
1) Методом getCatalog() делаем выгрузку справочника товарных групп; 
2) Методом getItems() получаем справочник товаров; 
3) Методом getItemsAvail() проходимся по нужным нам товарным подгруппам или товарам 
для получения доступного количества и цен.

Важно! При переносе резервов создается новый заказ; в него можно перенести резервы

Перенос резервов в новый заказ предполагает работу с пачками команд. Номер коман
ды заголовка пачки соответствует номеру команды отдельной строки. 
 
Для проверки результата переноса необходимо использовать метод 
getCommandResultDetails(). 
 
В случае ошибки по одному или нескольким товарам будет выполнен перенос товаров, 
кроме ошибочных. Статус обработки заголовочной команды пачки (у которой выходное 
значение OperationLineNo = 0 в методе getCommandResultDetails()) при этом будет 
"Ошибка", и в поле ProcessingResultComment будет указан результат обработки всей 
пачки: сколько строк (товаров) было обработано всего и сколько с ошибкой. 
Результат обработки каждой строки будет указан отдельно в поле ProcessingResultCom
ment, при этом в строках со статусом «Ошибка» будет указана причина ошибки перено
са. 
 
Для переноса резервов в новый заказ необходимо: 
 
1) Отправить команду setMoveOrderLineCommand() для переноса с параметрами: 
document_no – номер документа, откуда будет перенесен резерв; item_no – 
номер товара для переноса в новый заказ; qty – количество товара для переноса; 
operation_no – оставить пустым. 
 
2) Отправить команду setMoveOrderLineCommand() с параметрами: 
document_no – номер документа, откуда будет перенесен резерв; item_no – 
номер товара для переноса в новый заказ; qty – количество товара для переноса;

Результат выполнения команд при завершении с ошибкой: 
27 
Если при выполнении нескольких последовательных операций одна из них завершилась 
с ошибкой, это повлияет только на последующие операции, которые так или иначе свя
заны с ошибочной. Если последующие операции не связаны с ошибочной, они выполнят
ся в обычном режиме. 
Например: есть два неподтвержденных заказа, один вчерашний, второй - сегодняшний. 
Хотим добавить в них товары, но предварительно изменить дату вчерашнего заказа на 
текущую. Предположим, при изменении даты первого заказа мы ошиблись, операция за
вершилась с ошибкой, и дата осталась вчерашней. А потом мы сразу стали добавлять в 
эти два заказа товары. При этом в неизмененный вчерашний заказ товары не добавятся 
(поскольку дата должна быть корректной), а в сегодняшний товары добавятся успешно. 
Важно! Проверять результат обработки команды необходимо после каждой команды по 
работе с заказами. Необходимо запрашивать getCommandResult() (с разрешенной ча
стотой), пока не получите статус «Сделано» или «Ошибка». 
Успешное завершение выполнения команды означает только статус «Сделано». 
В случае получения статуса «Ошибка» необходимо проверить поле ErrorText (текст 
ошибки), при необходимости обратиться в тех.поддержку API. 
Остальные статусы являются промежуточными.

Web-сервис (API) B2B-модуля «Интернет-заказ» работает по протоколу SOAP (Simple Object Ac
cess Protocol) версии 1.1. поверх HTTPS. Обмен данными ведется посредством XML-сообщений, 
что позволяет реализовать удаленный вызов процедур (RPC - Remote Procedure Call). 
Для удобства работы с API размещается WSDL-описание (Web Services Description Language) вер
сии 1.1. — это язык описания Web-сервисов, основанный на XML. На WSDL описываются методы, 
входные и результирующие структуры данных, типы данных, сетевые адреса для обращения к 
сервису и т.д. 
WSDL-описание сервиса (с различными стилями WSDL) доступно по ссылкам: 
 https://api.merlion.com/dl/mlservice3?wsdl (стиль document/literal) 
 https://api.merlion.com/re/mlservice3?wsdl (стиль rpc/encoded) 
 https://api.merlion.com/rl/mlservice3?wsdl (стиль rpc/literal) 
Для тестирования API WSDL-описание тестового сервиса доступно по ссылкам: 
 https://apitest.merlion.com/dl/mlservice3?wsdl (стиль document/literal) 
 https://apitest.merlion.com/re/mlservice3?wsdl (стиль rpc/encoded) 
 https://apitest.merlion.com/rl/mlservice3?wsdl (стиль rpc/literal) 
По WSDL-описанию можно построить клиентский класс API, используя любую SOAP-библиотеку. 
Библиотека скачивает WSDL-описание, анализирует и формирует структуры данных, необходи
мые для вызова методов. Приложению остается заполнить структуры данными и выполнить за
прос. Информация из ответного XML-сообщения выдается в виде структуры, свойственной языку 
программирования. Все это избавляет приложение от необходимости обрабатывать XML. Суще
ствует возможность самостоятельно формировать SOAP-запросы в формате XML и разбирать от
веты, однако это трудоемкая задача, и работу с WSDL вручную выполнять нецелесообразно. 
Взаимодействие между серверами B2B-модуля «Интернет-заказ» и партнера осуществляется с 
помощью API. 
Сервис позволяет выполнять удаленные запросы к базе данных компании MERLION и получать 
актуальные данные, выполнять обновление в режиме он-лайн только необходимой части данных, 
в отличие от скачивания полного прайс-листа в формате XML, формируемого несколько раз в 
день.  
API позволяет:  
 Автоматически резервировать в системе товар на складе компании MERLION, а также по
лучать информацию по созданным и ранее отгруженным заказам; 
 Подписывать заказ на отгрузку без участия менеджера компании MERLION (при соблю
дении общих и финансовых условий); 
 Импортировать в учетную систему актуальные цены и доступное для резерва количество 
товаров на складе по интересуемым товарным группам; 
 Использовать функциональность сервиса в интернет-магазинах и других торговых пло
щадках для расширения товарного ассортимента и наполнения товарных карточек ха
рактеристиками и изображениями товаров. 
Работа с API осуществляется по HTTPS (Hypertext Transfer Protocol Secure) протоколу с исполь
зованием авторизационных данных B2B-модуля «Интернет-заказ». Параметры подключения к 
сервису зависят от используемого протокола. Подробную информацию и примеры подключения 
к API смотрите в разделе «Подключение к API».

Прайс-лист можно получить с помощью трех отдельных методов: 
 getCatalog() – для получения справочника товарных групп;  
 getItems() – для получения справочника товаров;  
 getItemsAvail() – для получения доступного количества товаров и цен. 
Такой подход позволяет получать актуальные данные по всему прайс-листу или только по инте
ресуемым товарным группам, обращаясь к каждому методу с разной периодичностью. 
Классификация в товарном справочнике построена по принципу: 
Направления ⇒ Группы ⇒ Подгруппы ⇒ Товары 
Направления содержат группы, группы содержат подгруппы, подгруппы содержат товары. То
вары могут находиться только в подгруппах. 
Для направлений, групп и подгрупп, содержащих в коде букву «А», она указывается кириллицей. 
На расчет доступного количества и цен влияют значения метода отгрузки и даты отгрузки. В 
зависимости от метода отгрузки определяется, по какому складу и по какому прайс-листу будет 
произведен расчет данных. От даты отгрузки зависит доступное количество товаров для резерва. 
ВАЖНО! При выборе доставочного метода отгрузки (например, "ДОСТАВКА") необходимо ука
зывать дату отгрузки, начиная с завтрашнего дня, т.к. на сегодня доставка не осуществляется 
(при этом доступное количество товара будет нулевое). 
Все данные по количеству товара возвращаются в штуках. 
Пример получения прайс-листа: 
1) Методом getCatalog() делаем выгрузку справочника товарных групп; 
2) Методом getItems() получаем справочник товаров; 
3) Методом getItemsAvail() проходимся по нужным нам товарным подгруппам или товарам 
для получения доступного количества и цен.

3.3 РАБОТА С ЗАКАЗАМИ 
20 
При работе с заказами необходимо знать следующее: 
 Заказы состоят из двух частей: заголовок и строки. 
 Все модификации заказов осуществляются через очередь команд. 
 Каждый метод для модификации заказов возвращает номер команды для последующего 
отслеживания результата обработки. 
 При резервировании товара указывается желаемое количество для резерва. В резерв 
ставится количество товара, доступное на текущий момент, которое может быть меньше 
желаемого в зависимости от параметров отгрузки заказа. 
 Проверить текущее состояние заказа можно при помощи методов getOrdersList() и 
getOrderLines(). 
 Учтенным заказ становится только после отгрузки товара со склада; до этого момента он 
считается неучтенным или учтенным частично, если отгружается несколькими частями 
(несколькими документами). 
Важно! Если выходным параметром метода является номер команды (return), то после выпол
нения команды необходимо сначала проверить результат ее выполнения с помощью метода 
getCommandResult().

Если при выполнении нескольких последовательных операций одна из них завершилась с 
ошибкой, это повлияет только на последующие операции, которые так или иначе связаны

обычном режиме. 
Например: есть два неподтвержденных заказа, один вчерашний, второй - сегодняшний.

вершилась с ошибкой, и дата осталась вчерашней. А потом мы сразу стали добавлять в 
эти два заказа товары. При этом в неизмененный вчерашний заказ товары не добавятся 
(поскольку дата должна быть корректной), а в сегодняшний товары добавятся успешно. 
Важно! Проверять результат обработки команды необходимо после каждой команды по 
работе с заказами. Необходимо запрашивать getCommandResult() (с разрешенной ча
стотой), пока не получите статус «Сделано» или «Ошибка». 
Успешное завершение выполнения команды означает только статус «Сделано». 
В случае получения статуса «Ошибка» необходимо проверить поле ErrorText (текст 
ошибки), при необходимости обратиться в тех.поддержку API. 
Остальные статусы являются промежуточными.

склада

1 ОСНОВНЫЕ СВЕДЕНИЯ 
3 
Web-сервис (API) B2B модуля «Гарантийное обслуживание» работает по протоколу SOAP (Sim
ple Object Access Protocol) версии 1.1. по HTTPS. Обмен данными ведется посредством XML
сообщений, что позволяет реализовать удаленный вызов процедур (RPC - Remote Procedure 
Call). 
Для удобства работы с API размещается WSDL-описание (Web Services Description Language) 
версии 1.1. — это язык описания Web-сервисов, основанный на XML. WSDL описывает методы, 
входные и результирующие структуры данных, типы данных, сетевые адреса для обращения к 
сервису и т.д. 
WSDL-описание сервиса (с различными стилями WSDL) доступно по ссылкам: 
 https://api.merlion.ru/dl/scservice?wsdl (стиль document/literal) 
 https://api.merlion.ru/re/scservice?wsdl (стиль rpc/encoded) 
 https://api.merlion.ru/rl/scservice?wsdl (стиль rpc/literal) 
Для тестирования API WSDL-описание тестового сервиса доступно по ссылкам: 
 https://apitest.merlion.ru/dl/scservice?wsdl (стиль document/literal) 
 https://apitest.merlion.ru/re/scservice?wsdl (стиль rpc/encoded) 
 https://apitest.merlion.ru/rl/scservice?wsdl (стиль rpc/literal) 
По WSDL-описанию, с помощью любой стандартной SOAP-библиотеки, можно автоматически 
построить клиентский класс API с таким же интерфейсом, как у серверного класса. При вызове 
метода клиентского класса происходит передача входных параметров и получение результиру
ющего набора данных в виде передачи через интернет XML-пакетов запроса (SOAP-Request) и 
ответа (SOAP-Response), которые SOAP-библиотекой «прозрачно» преобразуются в типовые 
структуры данных выходных параметров, согласно их объявлению. В случае добавления новых 
параметров или новых полей в серверном классе, в вашем клиентском классе они появятся ав
томатически, а вам останется только использовать эти данные по смыслу в вашей системе. 
Рекомендуем использовать именно этот подход. 
Существует возможность работать без использования WSDL, «вручную» создавая клиентский 
класс по техническому описанию. Обмен XML-пакетами между клиентским и серверным класса
ми будет так же «прозрачен», как и в первом случае с использованием WSDL, но при использо
вании новых возможностей серверного класса потребуется в первую очередь вручную изменить 
объявления методов в клиентском классе сервиса и только потом реализовать их использова
ние в своей системе. 
Рекомендуем использовать этот подход. 
Существует возможность самостоятельно формировать SOAP-запросы в формате XML и разби
рать SOAP-ответы от сервиса, однако это трудоемкая задача, т.к. потребует строгого соблюде
ния всех условий протокола, заложенных в стандартные SOAP-библиотеки. 
Такой подход категорически не рекомендуем использовать, т.к. теряется не только гиб
кость для расширения возможностей, но и надежность вашего сервиса. Простые доработки по 
добавлению параметра или поля будут приводить к краху работы сервиса, в отличие от исполь
зования стандартных библиотек. 
API SC позволяет импортировать данные по товарам, находящимся на гарантийном обслужи
вании.

Подключение к веб-сервису осуществляется по HTTPS протоколу с использованием протокола 
шифрования TLS версий 1.0, 1.1, 1.2 и Basic-аутентификации. 
Доступ к данным осуществляется с использованием авторизационных данных B2B-модуля 
b2b.merlion.com. В качестве логина необходимо передавать код клиента и логин, разделяя их 
символом вертикальной черты "|".  Пример: ABC0123|MYLOGIN. 
У используемого логина должны быть настроены права доступа к API SC, которые вы можете 
запросить в службе поддержки, оформив заявку на подключение на странице 
https://api.merlion.com или отправив письменный запрос на api@merlion.ru. 
ВАЖНО! Для работы с API требуется создать отдельный логин. Использование одного логина 
для работы под ним обычного пользователя и сервиса будет приводить к постоянным запросам 
авторизации для пользователя.
